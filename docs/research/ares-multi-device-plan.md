# ARES Multi-Device Architecture

## Vision
One ARES. All your devices. Same brain, different bodies.

## The Core Insight
The Mac Studio is already the droid brain — 24/7, running Hermes, MCP servers, the avatar renderer, voice pipeline. The iPhone, iPad, and Watch are **terminals**, not brains. They connect to the Studio, they don't host the agent.

This is different from every "AI companion" app today. They run the model on-device or cloud. ARES runs the brain on a dedicated machine — the terminal is just a thin client that feels like part of the robot.

## Architecture Layers

### Layer 0: The Brain (Mac Studio)
- Hermes agent (Discord-connected, fully autonomous)
- ARES API on :7860 (FastAPI, websocket + REST)
- MCP servers on :9501-:9520 (voice, perception, avatar, motion)
- Hermes Dashboard on :9119
- Ollama local model (optional, swap with cloud)
- **Source of truth** for all state

### Layer 1: Transport Protocol

```
Watch ──┬── WCSession (BLE) ──> iPhone ──┬── Tailscale/WiFi ──> Mac Studio
         └── Direct WiFi WS ───────> Mac Studio (when on home LAN)
```

Three tiers, auto-negotiated:

**Tier 1 (Preferred): Watch → iPhone → Mac Studio**
- Watch connects to iPhone via `WCSession` (Apple's native bluetooth relay)
- iPhone holds a persistent WebSocket to Mac Studio (wifi/tailscale)
- iPhone acts as proxy: forwards watch messages → Mac Studio, strips responses → watch
- Battery cost: watch negligible (BLE), iPhone ~5%/hr (background socket), Mac Studio zero

**Tier 2 (Watch alone on home WiFi): Watch → Mac Studio**
- Watch opens WebSocket directly to Mac Studio when on same LAN
- No iPhone needed — watch talks to UDP broadcast or hardcoded IP
- Battery cost: watch ~15-20%/hr (wifi + websocket)
- Falls back to Tier 1 when iPhone is nearby

**Tier 3 (Critical alerts): Push notification**
- Mac Studio sends push via APNs for urgent things (robot error, task complete)
- Uses Firebase Cloud Messaging or direct APNs with a provider cert
- Not for chat — only for "wake up" events

    │   ├── CaptionView.swift   # Caption overlay (shared)

### Layer 4: Data Sync Strategy

**Real-time (WebSocket):** Chat, state changes, captions, audio levels
- Mac Studio is the server
- Every device opens one WebSocket (or WCSession proxy)
- No history sync needed — the Studio holds the conversation

**Persistent (SwiftData + CloudKit):** Offline availability, boot new device
- Each device caches last N messages locally via SwiftData
- CloudKit private database syncs conversation summaries
- Only needed for: showing conversation on a device that just connected

### Layer 5: Watch App Design (Minimum Viable)

```
┌─────────────────────┐
│    ARES              │  Status bar: ● connected / ○ offline
│    ● Listening       │  Agent state + last caption
│                     │
│  "What's the plan    │  Voice dictation button
│   for today?"        │
│                     │
│  [🎤 Speak]          │  Big tap target
│                     │
│  Quick Replies:      │  5 context-sensitive buttons
│  [Yes] [No] [?]     │  Generated by brain based on last message
│  [Tell me more]      │
│  [Repeat that]       │
└─────────────────────┘
```

- Single-page app, no navigation
- Dictation uses `WKExtension.shared().visibleInterfaceController?.presentTextInputController`
- Quick replies are pushed by the brain via the transport
- Watch face complication shows connection status + last caption
