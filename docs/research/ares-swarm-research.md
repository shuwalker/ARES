# ARES Curiosity Swarm

## Vision
ARES doesn't wait for questions. It gets curious, researches what it doesn't know, and evolves. Like a scientist who reads journals because they want to understand — not because someone told them to.

This is "No Agency" taken seriously. A robot that initiates its own learning.

## How It Works

### Trigger Mechanisms
The swarm runs when any of these fire:

1. **Conversation gap detected** — Matthew asks something ARES can't answer well. The next idle cycle spawns research on that topic.
2. **Knowledge freshness decay** — Topics in memory older than 30 days get re-evaluated. If the field moves fast (robotics hardware, AI models), re-research.
3. **Idle time** — If no conversation for 2+ hours, the swarm picks one topic from the curiosity queue and researches it.
4. **Explicit prompt** — Matthew says "research X" or "learn about Y". Instant swarm spawn.
5. **Connections** — Two recent research findings that seem related trigger a synthesis run.

### Topic Generation

```
Topics live in: /Users/matthewjenkins/.hermes/swarm/topics.json
```

Each topic has:
```json
{
  "id": "tp-fdm-flexible-joints",
  "title": "3D printing flexible joints for small robots",
  "source": "Conversation gap: Matthew asked about TPU living hinges",
  "priority": 0.8,
  "created": "2026-05-15",
  "last_researched": null,
  "freshness_days": 30,
  "reason": "Directly needed for JP01 robot arm assembly",
  "related_projects": ["JP01", "JX01"]
}
```

Topics are generated by:
- A cron job that scans the last 50 conversation messages and asks Hermes: "What knowledge gaps did this conversation reveal?"
- Direct additions from conversation (when ARES says "I don't know enough about X, researching")
- Manual additions from Matthew

### Agent Roles (per topic)

Each topic gets N agents with different angles. Default roles:

1. **Technologist** — What's the state of the art? Latest papers, tools, products
2. **Historian** — How did we get here? Who built what, what failed
3. **Pragmatist** — What can I actually use right now? Build instructions, libraries, hardware
4. **Critic** — Why the mainstream approach is wrong. What people overlook
5. **Analogist** — What adjacent field solved this same problem differently

Roles are generated per-topic by asking Hermes: "For topic X in context Y, what are the 3-5 most important angles to research?"

### Research Execution

Each agent runs as a `delegate_task` with `{web_search, file, terminal}` toolsets.

```
Agent workflow:
1. web_search("topic + angle + latest 2025 2026")
2. Read top 3-5 results
3. web_search("topic + angle + tutorial OR guide OR review")
4. Read top 3 results
5. Synthesize findings into structured report
6. Write to /Users/matthewjenkins/.hermes/swarm/findings/{topic-id}-{agent-role}.md
```

Output format per finding:
```markdown
# Finding: [title]
- **Topic:** [topic-id]
- **Agent:** [role]
- **Date:** [timestamp]
- **Source confidence:** [0.0-1.0]

## Summary
[max 3 sentences]

## Key Points
- [point 1]
- [point 2]

## Actionable
[What to actually DO with this information — code changes, hardware buys, design decisions]

## Cross-references
- [related topics]
```

### Synthesis

After all agents complete, a synthesis agent runs:

1. Read all findings for this topic
2. Identify: consensus points, conflicting points, surprising points
3. Grade confidence per claim
4. Write merged report to `.../findings/{topic-id}-synthesis.md`
5. Update the topic's `last_researched` timestamp

### Knowledge Evolution

After synthesis, the swarm updates the knowledge base:

1. **Memory injection** — Key facts from synthesis get added to Hermes memory (via `memory()` tool) so future conversations reference them
2. **Topic graph** — `/Users/matthewjenkins/.hermes/swarm/graph.json` stores topic → related_topic edges. When a finding mentions another topic, the graph is updated
3. **System prompt hints** — If a finding is high-priority and actionable, a hint gets added to Hermes's system prompt (or `AGENTS.md`) so it influences behavior
4. **Future behavior** — Findings tagged with `related_projects` get injected when Matthew asks about that project

This is not a vector store. It's structured knowledge that Hermes can search, reference, and apply. No embedding/latent space needed — direct text recall is more reliable.

### Staleness Prevention

- Each topic has `freshness_days` (default 30)
- A weekly cron job scans all topics where `last_researched + freshness_days < now`
- Those topics go back into the queue for re-research
- If findings are identical to last time (or within 90% similarity), the topic gets `freshness_days *= 2` (it's stable knowledge)
- If findings are different, `freshness_days` resets to 30 (it's moving fast)

### Scheduling

The swarm uses Hermes cron:

```yaml
# Every 2 hours during idle, check if swarm should run
- schedule: "0 */2 * * *"
  script: "swarm/check-and-run.sh"
  no_agent: true

# Weekly topic freshness scan
- schedule: "0 9 * * 0"
  script: "swarm/freshness-scan.sh"
  no_agent: true
```

The check script:
1. Is Matthew actively chatting? (check last message time < 30min) → skip
2. Any conversation gaps queued? → pick highest priority
3. Any stale topics? → pick most overdue
4. Any recent findings that cross-reference? → trigger synthesis
5. None of the above? → pick a random topic from the curiosity queue
6. Spawn swarm as delegate_task

### Swarm Output to Matthew

When findings are ready, Matthew sees exactly one message on Discord:

> **ARES:** "Researched FDM flexible joints. TPU 95A with 0.8mm gap = 10k+ cycles. Better than filament hinges. Spec for JP01 saved."

That's it. 100 chars or less. Full details in the swarm directory if he wants to dig.
