# ARES Module Plan

How first-party productivity modules are built inside ARES-Desktop.

## Guiding rule

**Path A — build in ARES. Wrap third-party only if it gives 80%+ for free.**

Default to building the feature natively against Apple's frameworks (EventKit, SwiftData, etc.) so it lives inside ARES's own UI, memory, and approval layers. Only reach for a third-party app/integration (Notes.app via AppleScript, a SaaS API, etc.) when wrapping it delivers 80%+ of the value with materially less work than building it — and even then, keep the wrapper behind an ARES module interface.

## Module conventions

- Each module lives under `ARES-Desktop/Sources/ARES/Modules/<Name>/`.
- A `<Name>Module` is an `@MainActor final class … ObservableObject` holding state + the backing store; the `<Name>View` is the SwiftUI surface.
- Tabs are registered in the `ARESTab` enum (`App/ARESAppState.swift`) and routed in `Views/ARESRootView.swift`.
- Logging via `os.Logger(subsystem: "com.ares", category: "<Name>Module")`. No silent `catch {}`.

## Modules

### Calendar — `Modules/Calendar/`
- **Backing framework:** EventKit (`EKEventStore`, `EKEvent`).
- **Files:** `CalendarModule.swift`, `CalendarView.swift` (+ `AddEventSheet`).
- **Capability:** request full calendar access, list events for the next 7 days, add events.
- **Requires:** `NSCalendarsUsageDescription` + `NSCalendarsFullAccessUsageDescription` in Info.plist (generated by `ARES-Desktop/build-app.sh`).
- **Status:** ✅ Scaffolded — list + add working once access is granted.

### Tasks — `Modules/Tasks/`
- **Backing framework:** EventKit Reminders (`EKReminder`).
- **Files:** `TasksModule.swift`, `TasksView.swift` (+ `AddTaskSheet`).
- **Capability:** request full reminders access, list incomplete reminders sorted by due date, complete a reminder, add a task with optional due date.
- **Requires:** `NSRemindersUsageDescription` + `NSRemindersFullAccessUsageDescription` in Info.plist.
- **Status:** ✅ Scaffolded — list / complete / add working once access is granted.

### Notes — `Modules/Notes/`
- **Backing framework:** SwiftData (`@Model Note`). First SwiftData use in the app.
- **Files:** `Note.swift`, `NotesView.swift` (+ `NoteEditorView`).
- **Capability:** local persistent notes — create, edit (title/body), search, auto-`updatedAt`. Master/detail via `HSplitView`.
- **Requires:** `.modelContainer(for: Note.self)` on the `WindowGroup` in `App/ARESApp.swift`. No system permission.
- **Status:** ✅ Scaffolded — create / edit / search / persist working.

## Wiring checklist (per module)

1. Create `<Name>Module.swift` + `<Name>View.swift` under `Modules/<Name>/`.
2. Add a `case` to `ARESTab` (enum, `title`, `systemImage`) in `App/ARESAppState.swift`.
3. Add a routing `case` in `Views/ARESRootView.swift`.
4. Add any privacy usage-description keys to the Info.plist heredoc in `ARES-Desktop/build-app.sh`.
5. For SwiftData modules, register the model container in `App/ARESApp.swift`.

## Future modules (candidate, Path A first)

- **Mail / triage** — likely a wrapper (third-party value high), behind an ARES module interface.
- **Files / quick capture** — native (`FileManager` / SwiftData).
- **Contacts** — native EventKit-adjacent (`Contacts` framework); `NSContactsUsageDescription` already present.
